# PWA Best Practices & Checklist

> **What makes a PWA:** A web application that is installable, works offline, and behaves like a native app. The three pillars are: a **Web App Manifest**, a **Service Worker**, and **HTTPS**.
>
> **When to apply:** Any production web application — especially mobile-first apps, dashboards, tools, and content sites. Even if "full PWA" isn't the goal, the manifest + basic caching dramatically improves user experience.
>
> **Lighthouse PWA audit:** This file covers every item Lighthouse checks. Running `npx lighthouse --only-categories=pwa` will validate compliance.

### Manifest Field Reference

| Field | Required | Notes |
|---|---|---|
| `name` | ✅ | Full app name (max ~45 chars for install prompts) |
| `short_name` | ✅ | Name under icon on home screen (max ~12 chars) |
| `start_url` | ✅ | Entry point. Add `?source=pwa` to track installs in analytics. |
| `display` | ✅ | `standalone` (no browser chrome), `fullscreen`, `minimal-ui`, or `browser` |
| `icons` | ✅ | Minimum: 192×192 + 512×512. Include `maskable` variants. |
| `theme_color` | ✅ | Status bar / title bar color. Match your palette's `--bg` |
| `background_color` | ✅ | Splash screen background. Match `--bg`. |
| `scope` | Recommended | Restricts which URLs the PWA controls. Usually `/`. |
| `orientation` | Optional | `portrait-primary`, `landscape-primary`, `any` |
| `screenshots` | Recommended | Shown in "richer install UI" on Android/desktop Chrome |
| `shortcuts` | Optional | Quick actions from long-press on app icon |

### Maskable Icons

Maskable icons are cropped to safe zone (center 80%) by Android. Never put critical content outside the inner circle:

```
┌──────────────────────┐
│                      │
│    ┌──────────┐      │
│    │   SAFE   │      │  ← Only this area is guaranteed visible
│    │   ZONE   │      │     on all mask shapes (circle, squircle, etc.)
│    │  (80%)   │      │
│    └──────────┘      │
│                      │
└──────────────────────┘
```

Validate at: https://maskable.app/editor

### Light/Dark Theme Color Switching

```js
// Dynamically update theme-color meta tag when theme changes
function updateThemeColor(isDark) {
  const meta = document.querySelector('meta[name="theme-color"]');
  if (meta) meta.content = isDark ? '#0a0a1a' : '#f5f5f5';
}
```

## 3. Service Worker

### Registration (in `main.js` or entry point)

```js
// Register service worker — only in production, AFTER page load
if ('serviceWorker' in navigator && import.meta.env.PROD) {
  window.addEventListener('load', () => {
    navigator.serviceWorker
      .register('/sw.js', { scope: '/' })
      .then((reg) => {
        console.log('[SW] Registered:', reg.scope);

        // Check for updates periodically
        setInterval(() => reg.update(), 60 * 60 * 1000); // Hourly
      })
      .catch((err) => console.warn('[SW] Registration failed:', err));
  });
}
```

### Cache-First Strategy (for static assets)

Best for: CSS, JS bundles, fonts, images — assets that change only on deploy.

```js
// sw.js
const CACHE_NAME = 'static-v1';
const STATIC_ASSETS = [
  '/',
  '/css/design-tokens.css',
  '/css/components.css',
  '/js/main.js',
  '/offline.html',
];

// Install: pre-cache critical assets
self.addEventListener('install', (event) => {
  event.waitUntil(
    caches
      .open(CACHE_NAME)
      .then((cache) => cache.addAll(STATIC_ASSETS))
      .then(() => self.skipWaiting())
  );
});

// Activate: clean old caches
self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys().then((keys) =>
      Promise.all(
        keys
          .filter((key) => key !== CACHE_NAME)
          .map((key) => caches.delete(key))
      )
    ).then(() => self.clients.claim())
  );
});

// Fetch: cache-first for static, network-first for API
self.addEventListener('fetch', (event) => {
  const { request } = event;
  const url = new URL(request.url);

  // Skip non-GET requests
  if (request.method !== 'GET') return;

  // Skip cross-origin requests (analytics, etc.)
  if (url.origin !== self.location.origin) return;

  // API calls: network-first
  if (url.pathname.startsWith('/api/')) {
    event.respondWith(networkFirst(request));
    return;
  }

  // Everything else: cache-first
  event.respondWith(cacheFirst(request));
});
```

### Network-First Strategy (for API/dynamic data)

Best for: API responses, user data, frequently updated content.

```js
async function networkFirst(request) {
  const cache = await caches.open(CACHE_NAME);
  try {
    const response = await fetch(request);
    // Only cache successful responses
    if (response.ok) {
      cache.put(request, response.clone());
    }
    return response;
  } catch {
    // Network failed — try cache
    const cached = await cache.match(request);
    return cached || new Response('{"error":"offline"}', {
      status: 503,
      headers: { 'Content-Type': 'application/json' },
    });
  }
}
```

### Cache-First Strategy (for static assets)

```js
async function cacheFirst(request) {
  const cached = await caches.match(request);
  if (cached) return cached;

  try {
    const response = await fetch(request);
    if (response.ok) {
      const cache = await caches.open(CACHE_NAME);
      cache.put(request, response.clone());
    }
    return response;
  } catch {
    // Both cache and network failed — return offline page
    if (request.headers.get('Accept')?.includes('text/html')) {
      return caches.match('/offline.html');
    }
    return new Response('Offline', { status: 503 });
  }
}
```

### Stale-While-Revalidate Strategy

Best for: content that should show instantly but stay fresh — blog posts, product pages.

```js
async function staleWhileRevalidate(request) {
  const cache = await caches.open(CACHE_NAME);
  const cached = await cache.match(request);

  // Fetch in background regardless
  const fetchPromise = fetch(request)
    .then((response) => {
      if (response.ok) cache.put(request, response.clone());
      return response;
    })
    .catch(() => null);

  // Return cached immediately if available, or wait for network
  return cached || fetchPromise || new Response('Offline', { status: 503 });
}
```

### Strategy Decision Guide

| Content Type | Strategy | Why |
|---|---|---|
| App shell (HTML, CSS, JS) | Cache-first | Instant load; update on next visit |
| Fonts, icons, images | Cache-first | Rarely change; large payload |
| API responses | Network-first | Need fresh data; cache is fallback |
| Blog/CMS content | Stale-while-revalidate | Show fast, update in background |
| Auth tokens, payments | Network-only | Never cache sensitive data |
| Analytics, tracking | Network-only | Fire-and-forget; don't cache |


**Rules for offline pages:**
- All CSS must be inlined — no external stylesheets
- No external dependencies (no Google Fonts, no CDN scripts)
- Use system fonts only
- Keep it under 15 KB total
- Match the app's visual tone (use the same background and accent colors)

## 5. Install Prompt UX

### Deferred Install Prompt (Chrome/Edge)

```js
// capture-install.js
let deferredPrompt = null;
const installBanner = document.getElementById('install-banner');

// 1. Capture the event — Chrome fires this when installability criteria are met
window.addEventListener('beforeinstallprompt', (e) => {
  e.preventDefault();
  deferredPrompt = e;

  // Show YOUR custom install UI (not the browser's default)
  if (installBanner) installBanner.hidden = false;
});

// 2. When user clicks your install button
document.getElementById('install-btn')?.addEventListener('click', async () => {
  if (!deferredPrompt) return;

  deferredPrompt.prompt(); // Show native install dialog
  const { outcome } = await deferredPrompt.userChoice;

  console.log(`[PWA] Install ${outcome}`); // 'accepted' or 'dismissed'
  deferredPrompt = null;
  if (installBanner) installBanner.hidden = true;
});

// 3. Track successful installs
window.addEventListener('appinstalled', () => {
  console.log('[PWA] App installed');
  deferredPrompt = null;
  if (installBanner) installBanner.hidden = true;
  // Track in analytics
});
```

### When to Show the Prompt

```
❌ NEVER: Immediately on first visit (user has no context)
❌ NEVER: Repeatedly after dismissal
✅ GOOD:  After the user has engaged (viewed 3+ pages, or used a feature)
✅ GOOD:  After a successful action (saved something, completed a task)
✅ GOOD:  On a "settings" page as a persistent but non-intrusive option
```

```js
// Show install prompt after engagement threshold
const pageViews = parseInt(sessionStorage.getItem('pv') || '0', 10) + 1;
sessionStorage.setItem('pv', String(pageViews));

if (pageViews >= 3 && deferredPrompt && !localStorage.getItem('install-dismissed')) {
  installBanner.hidden = false;
}

// Track dismissal to avoid nagging
document.getElementById('install-dismiss')?.addEventListener('click', () => {
  installBanner.hidden = true;
  localStorage.setItem('install-dismissed', 'true');
});
```

## 6. Update Notification

When a new service worker version is available, notify the user:

```js
// In main.js — after registration
navigator.serviceWorker.register('/sw.js').then((reg) => {
  reg.addEventListener('updatefound', () => {
    const newWorker = reg.installing;
    if (!newWorker) return;

    newWorker.addEventListener('statechange', () => {
      if (newWorker.state === 'installed' && navigator.serviceWorker.controller) {
        // New version available — show update toast
        showUpdateNotification();
      }
    });
  });
});

function showUpdateNotification() {
  const toast = document.createElement('div');
  toast.className = 'update-toast';
  toast.setAttribute('role', 'alert');
  toast.innerHTML = `
    <p>A new version is available.</p>
    <button onclick="window.location.reload()" class="btn btn--primary btn--sm">Update</button>
  `;
  document.body.appendChild(toast);
}
```

## 7. Display Mode CSS

Style differently when running as installed PWA vs. in browser:

```css
/* Standalone (installed PWA) — no browser chrome */
@media (display-mode: standalone) {
  .site-header { padding-top: env(safe-area-inset-top, 0px); }
  .browser-only { display: none; }  /* Hide "install app" promos */
}

/* In-browser — show install prompts */
@media (display-mode: browser) {
  .pwa-only { display: none; }
}

/* Picture-in-picture, window controls overlay, etc. */
@media (display-mode: minimal-ui) {
  /* Minimal chrome styles */
}
```

```js
// Detect display mode in JavaScript
const isStandalone =
  window.matchMedia('(display-mode: standalone)').matches ||
  ('standalone' in navigator && navigator.standalone);

if (isStandalone) {
  document.documentElement.classList.add('is-pwa');
  // Skip "install" prompts, adjust navigation, etc.
}
```

## 9. Lighthouse PWA Audit Checklist

These are the exact items Lighthouse checks. All must pass:

| Check | Requirement | Covered In |
|---|---|---|
| **Installable** | Valid manifest with `name`, `icons`, `start_url`, `display` | § 1 Manifest |
| **192px icon** | At least one 192×192 PNG icon | § 1 Icons |
| **512px icon** | At least one 512×512 PNG icon | § 1 Icons |
| **Maskable icon** | At least one icon with `purpose: "maskable"` | § 1 Maskable |
| **Sets theme-color** | `<meta name="theme-color">` in HTML | § 2 Meta Tags |
| **Viewport meta** | `<meta name="viewport" content="width=device-width, initial-scale=1">` | § 2 Meta Tags |
| **Service worker** | Registered SW that controls the page | § 3 Registration |
| **Offline fallback** | Returns 200 when offline (at least cached shell or offline page) | § 4 Offline |
| **start_url responds** | `start_url` returns 200 when offline | § 3 Pre-cache |
| **HTTPS** | Served over HTTPS (or localhost for dev) | Deployment |
| **Redirects HTTP → HTTPS** | All HTTP requests redirect to HTTPS | Deployment |
| **Splash screen** | `background_color`, `theme_color`, `name`, 512px icon all set | § 1 Manifest |
| **Content sized to viewport** | No horizontal overflow on mobile | mobile-responsive.md |
| **Has `<meta name="description">`** | SEO description present | § 2 Meta Tags |

## 10. Security Requirements

PWAs have strict security requirements:

```
✅ HTTPS required (except localhost)
✅ All resources loaded over HTTPS (no mixed content)
✅ Service worker scope matches manifest scope
✅ CSP headers configured (Content-Security-Policy)
❌ Never cache auth tokens or sensitive data in service worker
❌ Never cache POST responses
❌ Never serve stale data for payment or authentication flows
```

### Content Security Policy Header

```
Content-Security-Policy:
  default-src 'self';
  script-src 'self';
  style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;
  font-src 'self' https://fonts.gstatic.com;
  img-src 'self' data: https:;
  connect-src 'self' https://api.yourdomain.com;
  manifest-src 'self';
  worker-src 'self';
```
